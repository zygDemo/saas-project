<a id="1-项目概述"></a>
## 1. 项目概述

<a id="11-项目背景"></a>
### 1.1 项目背景

予艺助手是一款基于 uni-app 3 + Vue 3 + TypeScript 开发的多端移动应用，工程范围明确限定为 `D:\zygProject\GitHub\saas-project\saas-mobile`。本项目服务于车贷 SaaS 平台的一线移动场景，核心目标是让业务员/客户经理在手机端完成获客、扫码、资料采集、进件提交、补件跟进、签约跟进、订单查看和典当办理。

本 PRD 聚焦 `saas-mobile` 移动端项目，不替代 `saas-web` 管理后台 PRD，也不展开 `saas-api` 后端管理能力设计。涉及后台管理、审批配置、产品配置、资方配置、组织用户管理等能力，仅作为移动端使用前提或接口依赖描述。

<a id="15-支持平台与端能力边界"></a>
### 1.5 支持平台与端能力边界

| 平台 | 首期要求 | 说明 |
|------|----------|------|
| H5 | 必须支持 | 用于浏览器、微信内置浏览器、业务员分享链接和客户扫码办理 |
| 微信小程序 | 必须支持 | 作为业务员移动办公和客户扫码办理的重点小程序端 |
| 支付宝/百度/抖音/QQ/快手/飞书/钉钉等小程序 | 兼容设计 | 以 uni-app 能力为基础，首期不强制逐端上线验收 |
| Android / iOS App | 后续适配 | 业务代码需避免明显 H5-only 写法，便于后续 App 打包 |
| HarmonyOS | 后续适配 | 作为 uni-app 可适配方向保留，不作为首期上线阻塞项 |

多端约束：

1. 页面布局统一使用 rpx、uni-app 生命周期和条件编译，避免直接依赖浏览器专属 API。
2. 二维码、文件上传、拍照/OCR、拨号、定位等能力必须考虑 H5 与小程序差异。
3. 登录态、机构上下文、客户扫码 session token 需要在多端存储方案下保持一致。
4. 首期验收优先级为 H5 > 微信小程序 > 其他小程序/App。

<a id="2-产品定位与目标"></a>
## 2. 产品定位与目标

<a id="21-产品定位"></a>
### 2.1 产品定位

CarLoanH5 / saas-mobile 定位为车贷 SaaS 平台的多端移动业务端，核心是“业务员移动办公”。产品面向汽车金融中介机构、贷款服务机构、区域车贷团队的一线人员，提供移动化获客、资料采集、进件提交、补件处理、签约跟进、订单查看、典当办理等能力；同时通过 H5/小程序二维码链路承载客户自助资料填写、补件和签约。

<a id="22-产品目标"></a>
### 2.2 产品目标

1. 支持多机构、多部门、多业务员的移动办公，所有业务数据具备机构维度和人员归属。
2. 打通线索、客户、车辆、进件、审批、签约、放款、订单和贷后提醒主流程。
3. 支持客户扫码进件：客户通过业务员专属二维码进入资料填写、授权、文件上传、签约和进度查看。
4. 保留现有 CarLoanH5 页面资产，逐步替换旧接口和 mock 数据，向 SaaS 后端统一接口规范对齐。
5. 强化移动端待办中心、业务统计、权限控制、文件上传、补件、签约通知能力。

<a id="23-核心价值"></a>
### 2.3 核心价值

| 价值 | 描述 |
|------|------|
| 移动化办公 | 业务员随时录线索、提进件、处理补件、跟进签约和订单 |
| 标准化流程 | 按产品资料清单和审批流程驱动，降低漏填、错填、重复提交 |
| SaaS 多租户 | 支持机构级数据隔离、机构维度配置和按量计费扩展 |
| 客户自助办理 | 扫码完成授权、资料上传、补件和签约，减少业务员重复录入 |
| 数据驱动 | 业务员可查看线索、进件、审批、签约、放款等关键指标和待办 |
| 风控合规 | 客户授权、敏感信息脱敏、第三方查询留痕、审批留痕 |

<a id="24-成功指标kpi"></a>
### 2.4 成功指标（KPI）

| 指标 | 目标口径 |
|------|----------|
| 进件资料一次性完整率 | 提交即满足产品资料清单的进件数 / 总提交进件数 |
| 线索转进件率 | 已转进件线索数 / 有效线索数 |
| 进件到初审平均时长 | 初审完成时间 - 提交时间 |
| 补件平均处理时长 | 补件完成时间 - 补件发起时间 |
| 移动端业务员日活率 | 日活业务员 / 已启用业务员 |
| 签约完成率 | 签约完成进件数 / 审批通过进件数 |
| 放款转化率 | 已放款订单数 / 已提交进件数 |

<a id="32-本期不包含或二期开启"></a>
### 3.2 本期不包含或二期开启

- 完整催收工作台、法务诉讼管理。
- 复杂 BPMN 流程引擎；首期可用配置化审批节点和状态流转实现。
- 高级 AI 风控模型；首期保留规则引擎和第三方风控查询接口。
- 第三方支付自动代扣；首期可保留还款登记和支付通道适配接口。
- GPS 位置实时监控大屏；首期支持 GPS 安装登记和设备信息保存。
- 多语言完整运营内容；移动端目前以中文业务流程为主。

<a id="33-业务假设"></a>
### 3.3 业务假设

1. 平台方开通租户机构，机构下创建部门和用户。
2. 每个客户进件必须绑定机构、业务员和来源渠道，方便数据隔离和业绩归属。
3. 不同机构可配置不同产品、资料清单、审批流程、资方和第三方接口开关。
4. 第三方接口可能失败或产生费用，系统必须有调用日志、幂等保护和人工兜底路径。
5. 客户扫码进件的二维码参数必须具备签名或短码化机制，避免篡改业务员归属。

<a id="4-用户角色与权限"></a>
## 4. 用户角色与权限

<a id="41-角色体系"></a>
### 4.1 角色体系

| 角色 | 使用端 | 核心权限 |
|------|--------|----------|
| 平台超级管理员 | 管理后台 | 开通机构、套餐、全局配置、跨机构查看 |
| 机构管理员 | 管理后台 | 管理本机构部门、用户、角色、产品、业务数据 |
| 部门经理 | 管理后台/移动端 | 查看本部门及下级团队数据、分配线索、审批或督办 |
| 业务员/客户经理 | 移动端 | 线索跟进、客户资料、车辆资料、提交进件、处理补件、跟进签约 |
| 审批员/风控 | 管理后台/移动端可选 | 审批进件、发起补件、查看风控结果和审批记录 |
| 财务 | 管理后台/移动端可选 | 放款确认、还款登记、财务凭证、订单查看 |
| 客服/催收 | 管理后台/移动端可选 | 还款提醒、逾期跟进、联系记录 |
| 客户 | 客户扫码 H5/小程序 | 本人资料填写、授权、文件上传、补件、签约、进度查看 |

<a id="42-移动端权限控制要求"></a>
### 4.2 移动端权限控制要求

1. 登录后保存 `orgId`、`deptId`、`roles/roleKeys`、`permissions` 到本地持久化状态。
2. 工作台入口按角色或权限隐藏/禁用，例如审批入口仅审批员/经理可见，财务入口仅财务可见。
3. 所有接口请求必须携带 Token，并在可能时携带 `X-Org-Id`，由后端再次校验机构数据隔离。
4. 客户扫码端使用短期 Token，不等同后台用户或业务员 Token。
5. 敏感信息完整查看、导出、下载合同、查看风控报告需要具备对应权限并记录日志。

<a id="43-权限矩阵移动端核心"></a>
### 4.3 权限矩阵（移动端核心）

| 查看线索 | 本人 | 本部门 | 否 | 否 | 否 |

| 签约跟进 | 是 | 是 | 查看 | 查看 | 本人签署 |

<a id="5-核心业务流程"></a>
## 5. 核心业务流程

<a id="51-主业务流程"></a>
### 5.1 主业务流程

```text
线索获取
  ↓
线索分配 / 业务员跟进
  ↓
预审进件提交（身份证 OCR、行驶证 OCR、申请金额、期数、授权书）
  ↓
客户资料采集（主借人、联系人、车辆、银行卡、文件）
  ↓
选择产品 / 资方推荐
  ↓
提交资料校验与初审
  ↓
初审：通过 / 拒绝 / 补件
  ↓
终审或资方审批：通过 / 拒绝 / 补件 / 等待
  ↓
合同生成 / 电子签 / 视频面签
  ↓
GPS 安装 / 抵押办理 / 出账申请
  ↓
放款确认 / 生成订单和还款计划
  ↓
还款提醒 / 还款登记 / 逾期管理
  ↓
结清 / 解押 / 归档
```

<a id="52-线索到进件流程"></a>
### 5.2 线索到进件流程

| 步骤 | 触发人 | 系统动作 | 结果 |
|------|--------|----------|------|
| 新建线索 | 业务员/管理员/外部接口 | 校验手机号是否重复 | 生成线索 |
| 分配线索 | 管理员/部门经理/系统 | 按规则分配或抢单 | 绑定业务员 |
| 跟进线索 | 业务员 | 记录电话、微信、到店、意向 | 更新线索状态 |
| 转为客户 | 业务员 | 创建客户档案 | 线索状态变为已转化 |
| 发起进件 | 业务员/客户扫码 | 选择产品并采集资料 | 生成进件草稿 |
| 提交进件 | 业务员/客户 | 资料校验、文件清单校验 | 进入待初审/预审 |

<a id="53-审批流程"></a>
### 5.3 审批流程

| 阶段 | 处理人/系统 | 动作 | 可流转状态 |
|------|-------------|------|------------|
| 资料校验 | 系统 | 必填项、文件清单、图片质量、授权记录校验 | 通过 / 待补件 |
| 预审 | 系统/风控 | 规则引擎、大数据查询、车辆估值 | 建议通过 / 拒绝 / 人工复核 |
| 初审 | 审批员 | 核验资料真实性和完整性 | 通过 / 拒绝 / 补件 |
| 终审 | 高级审批员/经理 | 核定额度、期限、利率、资方 | 通过 / 拒绝 / 补件 |
| 资方审批 | 接口/人工 | 提交资方系统或人工录入结果 | 通过 / 拒绝 / 补件 / 等待 |

<a id="54-补件流程"></a>
### 5.4 补件流程

补件任务必须包含：补件原因、补件项、截止时间、处理人、处理状态、提交时间、审核结果。移动端应支持待补件提醒、补件详情、按资料项上传、补件完成提交。

| 场景 | 发起人 | 要求 |
|------|--------|------|
| 资料缺失 | 系统/审批员 | 指定缺失资料项，业务员或客户补充上传 |
| 图片不清晰 | 审批员 | 标注需重拍文件，原文件保留历史版本 |
| 信息不一致 | 审批员/风控 | 指定字段修改原因，修改后留痕 |
| 资方要求补充 | 资方/审批员 | 记录资方反馈并形成补件任务 |

<a id="55-签约与放款流程"></a>
### 5.5 签约与放款流程

1. 审批通过后生成合同和签约任务。
2. 客户完成电子合同签署，业务员可在移动端跟进签约状态。
3. 如需要视频面签，系统发起面签并保存面签记录、录屏或凭证。
4. 财务/业务员登记 GPS 安装信息和抵押办理信息。
5. 财务确认放款，系统生成订单、还款计划和放款流水。
6. 放款后进件状态归档，订单进入贷后管理。

<a id="8-客户扫码进件端要求"></a>
## 8. 客户扫码进件端要求

<a id="81-场景说明"></a>
### 8.1 场景说明

客户通过业务员专属二维码进入 H5/小程序，完成基础信息填写、授权、资料上传、签约和进度查看。二维码需包含或可解析出 `orgId`、`salesmanId`、`channelCode`、`productId`、`businessType`、过期时间和签名信息。

<a id="82-功能清单"></a>
### 8.2 功能清单

| 模块 | 功能 | 说明 |
|------|------|------|
| 入口识别 | 二维码参数解析 | 识别机构、业务员、渠道、产品、业务类型 |
| 手机验证 | 短信验证码登录 | 防止冒用和资料错绑 |
| 授权确认 | 隐私协议、征信授权、电子签授权 | 记录授权时间、IP、设备、版本 |
| 资料填写 | 身份、联系人、车辆、银行卡、工作/收入 | 支持 OCR 自动填充 |
| 文件上传 | 身份证、行驶证、银行卡、车辆照片等 | 根据产品文件清单动态展示 |
| 进度查看 | 查看申请状态、补件要求、签约入口 | 只展示本人申请 |
| 合同签署 | 跳转电子签 | 接入 e签宝等服务 |
| 补件处理 | 按补件任务补充资料 | 补件完成通知业务员/审批员 |

<a id="83-安全要求"></a>
### 8.3 安全要求

1. 客户端访问使用短期 Token，不等同后台用户 Token。
2. 客户只能查看本人申请，不允许通过 ID 枚举访问他人数据。
3. 二维码参数需签名或短码化，避免篡改业务员归属。
4. 客户上传文件需进行类型、大小、病毒/恶意文件基础校验。
5. 授权协议版本变更后，需重新记录客户确认。

<a id="106-关键业务规则"></a>
### 10.6 关键业务规则

1. 同一机构内，同一身份证号同一时间只能存在一个未完结进件，可配置是否允许并行。
2. 未记录客户授权时，不允许调用征信、大数据、电子签等敏感服务。
3. 审批通过后的核心字段变更（金额、期限、利率、资方）必须重新审批或记录特批。
4. 放款确认后，进件不可回退到审批环节，只能通过冲正/作废流程处理。
5. 还款计划生成后，如调整金额或期限，必须生成调整记录并保留旧计划。
6. 敏感数据查看、导出、下载合同和征信报告必须记录日志。
7. 第三方接口按次计费时，重复点击需要幂等保护，避免重复扣费。

<a id="114-接口安全"></a>
### 11.4 接口安全

1. 后台和移动端登录态接口需携带 `Authorization: Bearer ***`。
2. 为兼容旧移动端，短期内继续兼容 `Authorization: Bearer ***`。
3. Token 有效期默认 2 小时，RefreshToken 有效期默认 7 天。
4. 敏感接口增加签名验证、二次确认或短信验证码。
5. 接口限流：单用户默认 100 次/分钟，登录和短信接口单独限流。
6. 所有写操作需记录 `requestId`，支持幂等控制。
7. 文件上传接口需校验文件类型、大小、来源和权限。

<a id="16-风险与待确认事项"></a>
## 16. 风险与待确认事项

| 风险/问题 | 影响 | 处理建议 |
|-----------|------|----------|
| 新旧接口前缀不统一 | 前端适配成本增加 | 后端网关兼容 `/m`，中期统一 `/api/m` |
| 登录返回结构不一致 | 登录态保存失败 | 前端做兼容解析，后端输出统一规范 |
| 第三方接口费用和失败 | 影响成本与流程稳定 | 接口开关、调用日志、重试和人工兜底 |
| 二维码参数可篡改 | 业务员归属错误、越权 | 短码化、签名、过期时间、服务端校验 |
| 多租户隔离遗漏 | 数据安全事故 | 后端强制 orgId 过滤，前端仅作为展示上下文 |
| 敏感信息合规 | 隐私风险 | 脱敏、授权记录、操作日志、最小化采集 |
| 文件上传稳定性 | 补件/进件失败 | 压缩、重试、进度、失败恢复 |

待确认：

1. SaaS 后端最终移动端接口前缀采用 `/m`、`/api/m` 还是网关双兼容。
2. 登录接口返回字段是否统一为 `{ token, refreshToken, expires, userInfo }`。
3. 业务员二维码是否由后端生成短码，还是前端临时生成长链接二维码。
4. 客户扫码端是否继续复用现有 `pages/business/*`，还是拆分独立 `pages/client/*`。
5. 各机构产品、资料清单、审批流程的最小配置范围。
6. 电子签、OCR、车300、短信等第三方服务商最终选型和计费方式。
